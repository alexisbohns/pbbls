# iOS Auth Views Redesign ÔÇö Issue #360

**Date:** 2026-05-01
**Issue:** [#360 ÔÇö [Feat] Improve login and signup views in iOS](https://github.com/Bohns/pbbls/issues/360)
**Milestone:** M29 ┬Ě Extended login and signup on iOS
**Scope:** Pixel-perfect overhaul of `apps/ios/Pebbles/Features/Auth/AuthView.swift` (login + signup) plus extraction of shared primitives reused by both Welcome and Auth flows.

## Context

`WelcomeView` was recently overhauled (#358) with an animated splash and polished landing. The auth views it pushes into (`AuthView` for login and signup) were not part of that pass and still render the system `Picker(.segmented)`, `roundedBorder` text fields, blue checkboxes, and a default `borderedProminent` button. This PR brings those screens to parity with the new visual language and factors out the components that the welcome and auth flows share so future entry points can drop them in for free.

## Goals

- Replace the auth-screen visuals so login and signup match the mockups in issue #360 at the default Dynamic Type size on a 390pt iPhone.
- Add Apple/Google sign-in buttons and the legal disclaimer to the auth screens; make them visually identical to the buttons on `WelcomeView`.
- Extract shared primitives (`AppleSignInButton`, `GoogleSignInButton`, `LegalDisclaimerText`, `PebblesPrimaryButtonStyle`, `PebblesTextInput`, `PebblesCheckbox`, `PebblesAuthSwitcher`) and have BOTH `WelcomeView` and the new `AuthView` consume them.
- Stay within the project's existing data flow ÔÇö no new state in `AuthView`, no changes to `SupabaseService`, no new error UI.
- Comply with Apple HIG (44pt tap target minimum) and respect Dynamic Type so the layout scales correctly at accessibility text sizes.

## Non-goals

- Snapshot / pixel-diff testing.
- New error treatments or retry UI.
- Animation polish on switcher mode change (default is fine).
- Touching the welcome carousel's two top buttons (`Create an account`, `Log in`) ÔÇö they're a different visual variant only used there.
- Touching `LegalDocumentSheet`, the `LegalDoc` enum, or any `SupabaseService` wiring.

## Component contracts

All sizes use `@ScaledMetric(relativeTo: .body)` so they scale with Dynamic Type. The default values listed below are pixel-perfect at the default Dynamic Type size on a 390pt iPhone (i.e. what the mockups show).

### `PebblesTextInput`

```swift
struct PebblesTextInput: View {
    let placeholder: LocalizedStringResource
    @Binding var text: String
    var isSecure: Bool = false
    var contentType: UITextContentType? = nil
    var keyboard: UIKeyboardType = .default
    var autocapitalization: TextInputAutocapitalization = .sentences
    var autocorrection: Bool = true
}
```

- Shape: `RoundedRectangle(cornerRadius: 12)`, white fill, 1pt `pebblesBorder` stroke.
- Height: `.frame(minHeight: 52)` (scales).
- Internal padding: 16pt horizontal (scales).
- Text and placeholder colors are both `pebblesMutedForeground`. Achieved with `.foregroundStyle(pebblesMutedForeground)` on the field and a custom placeholder overlay (SwiftUI's default placeholder uses `.secondary` and isn't directly recolorable on iOS 17 without an overlay).
- Font: `.body` (scales).
- Branches internally on `isSecure` to render a `TextField` or `SecureField`; applies `.textContentType`, `.keyboardType`, `.textInputAutocapitalization`, `.autocorrectionDisabled` from props.

### `PebblesCheckbox`

```swift
struct PebblesCheckbox: View {
    @Binding var isChecked: Bool
    let prefix: LocalizedStringResource    // "I accept the "
    let linkText: LocalizedStringResource  // "Terms of Service"
    let onLinkTap: () -> Void
}
```

- Visual box: 28├Ś28pt `RoundedRectangle(cornerRadius: 12)` (scales).
  - Empty: white fill, 1pt `pebblesBorder` stroke. Inner glyph `Image(systemName: "square")` in `pebblesMutedForeground`, `.font(.body)`.
  - Checked: `pebblesAccent` fill and stroke. Inner glyph `Image(systemName: "checkmark.square")` in `pebblesBackground`, `.font(.body)`.
- Tappable area: `.frame(minWidth: 44, minHeight: 44)` `.contentShape(Rectangle())` ÔÇö guarantees the HIG 44pt tap minimum even though the visual box is 28pt.
- Label: `(Text(prefix) + Text(linkText).underline().foregroundColor(.pebblesAccent))`, base color `pebblesMutedForeground`, `.font(.subheadline)`. Whole-label tap fires `onLinkTap()` (same simplification as the existing `ConsentCheckbox`).
- Layout: `HStack(alignment: .center, spacing: 12)`.
- Accessibility: `.accessibilityElement(children: .combine)` to merge the box and label into a single element. Traits: `.isButton`, plus `.isSelected` when `isChecked`. `accessibilityLabel` describes the consent statement; the link is an `accessibilityAction(named: "Open <linkText>", action: onLinkTap)`.

### `PebblesPrimaryButtonStyle: ButtonStyle`

```swift
struct PebblesPrimaryButtonStyle: ButtonStyle {
    var isLoading: Bool = false
}
```

- Shape: `Capsule()`. Height: `.frame(minHeight: 52)` (scales). Full width.
- Enabled (`\.isEnabled == true`): fill `pebblesAccent`, label `.white` `.medium`.
- Disabled: fill `pebblesMuted`, label `pebblesBorder`.
- `isLoading == true`: replace label with `ProgressView().tint(.white)`. Caller is expected to also pass `.disabled(true)` so the press isn't re-fired.
- Press feedback: `configuration.isPressed ? 0.85 : 1.0` opacity.
- Reads `\.isEnabled` from the environment.

### `AppleSignInButton`

```swift
struct AppleSignInButton: View {
    var action: () -> Void
}
```

- `Button(action: action)` with label `HStack(spacing: 8) { Image(systemName: "applelogo"); Text("Continue with Apple") }`, `.white` foreground, `.medium` weight.
- Capsule fill `.black` (light AND dark mode ÔÇö Apple brand requirement).
- Height `.frame(minHeight: 52)` (scales). Full width.

### `GoogleSignInButton`

```swift
struct GoogleSignInButton: View {
    var action: () -> Void
}
```

- Same shape and height as Apple.
- Capsule fill `.white`. 1pt `pebblesBorder` stroke.
- Label: `HStack(spacing: 8) { Image("GoogleGMark") (18├Ś18, scales); Text("Continue with Google") }`, foreground `pebblesForeground`, `.medium` weight.

### `PebblesAuthSwitcher`

```swift
struct PebblesAuthSwitcher: View {
    @Binding var mode: AuthView.Mode
}
```

- Renders `Picker("Mode", selection: $mode) { ForEach(Mode.allCases) ÔÇŽ }` with `.pickerStyle(.segmented)`, full content width.
- Colors are configured ONCE at app launch in `PebblesApp.init`:
  - `UISegmentedControl.appearance().backgroundColor = UIColor(named: "Muted")`
  - `UISegmentedControl.appearance().selectedSegmentTintColor = UIColor(named: "MutedForeground")`
  - selected title text attrs: `.white`, `.medium`
  - normal title text attrs: `UIColor(named: "MutedForeground")`, `.regular`
- These settings are global; the app currently has no other `.segmented` Picker, so global is safe. If/when another segmented Picker is added, scope via `appearance(whenContainedInInstancesOf:)`.
- `UISegmentedControl` scales its title text with Dynamic Type by default; we accept that the default control height (~32pt) is below the HIG 44pt minimum, the same exception Apple itself ships.

### `LegalDisclaimerText`

```swift
struct LegalDisclaimerText: View {
    var onTermsTap: () -> Void
    var onPrivacyTap: () -> Void
}
```

- Lifted verbatim from `WelcomeView.revealedContent`'s closing `Text(...)` and its `OpenURLAction` handler.
- Renders `Text("Read our [Terms](pebbles://legal/terms) and [Privacy](pebbles://legal/privacy) before creating an account with Apple or Google.")`, `.font(.caption)`, `pebblesMutedForeground`, `.tint(pebblesAccent)`, centered, `.fixedSize(horizontal: false, vertical: true)`.
- The `OpenURLAction` resolves `pebbles://legal/terms` Ôćĺ `onTermsTap()`, `pebbles://legal/privacy` Ôćĺ `onPrivacyTap()`.

## Data flow

The data wiring stays nearly identical to today's `AuthView`. The view-local state and `SupabaseService` calls don't change; only the validation predicates and the in-flight visual change.

### State

`AuthView` continues to own:

```swift
@State private var mode: Mode             // .login or .signup, set from initialMode
@State private var email = ""
@State private var password = ""
@State private var termsAccepted = false
@State private var privacyAccepted = false
@State private var isSubmitting = false
@State private var presentedLegalDoc: LegalDoc?
@Environment(SupabaseService.self) private var supabase
```

No new state. `isSubmitting` now drives BOTH `.disabled(true)` AND the `PebblesPrimaryButtonStyle(isLoading:)` parameter on the primary button.

### `canSubmit` predicate

Same logic as today, extracted into a static method on `AuthView` so it's unit-testable:

```swift
extension AuthView {
    static func canSubmit(
        mode: Mode,
        email: String,
        password: String,
        termsAccepted: Bool,
        privacyAccepted: Bool,
        isSubmitting: Bool
    ) -> Bool {
        guard !isSubmitting,
              email.contains("@"),
              password.count >= 6 else { return false }
        if mode == .signup { return termsAccepted && privacyAccepted }
        return true
    }
}
```

The view's computed `canSubmit` is a one-line call into this. `\.isEnabled` is set via `.disabled(!canSubmit)`, which `PebblesPrimaryButtonStyle` reads from the environment to switch between enabled and disabled colors automatically.

### Submit flow

```swift
private func submit() {
    isSubmitting = true
    Task {
        switch mode {
        case .login:  await supabase.signIn(email: email, password: password)
        case .signup: await supabase.signUp(email: email, password: password)
        }
        isSubmitting = false
    }
}
```

Same as today. `SupabaseService.signIn / signUp` already write `authError` on failure and flip `session` on success; `RootView` swaps to the tab bar on session change. No behavior change.

### OAuth flow

Apple/Google buttons in `AuthView` call `Task { await supabase.signInWithApple() }` / `signInWithGoogle()`, mirroring `WelcomeView`. They share the same `isSubmitting` flag with the email submit, so tapping Apple while a sign-up is in flight is blocked.

### Mode change cleanup

Same as today: `.onChange(of: mode)` clears `authError` and resets the consent checkboxes when switching back to login. `.onChange(of: email)` and `.onChange(of: password)` clear `authError` on edit.

### Error display

`supabase.authError` rendered as `Text(error)` in `.red`, between the form and the primary button. Same placement and visual as today ÔÇö no new error treatment.

### Welcome view changes (data flow)

- `runApple()` / `runGoogle()` stay in `WelcomeView`, called by `AppleSignInButton` and `GoogleSignInButton`'s `action` closures.
- `LegalDisclaimerText` takes `onTermsTap` / `onPrivacyTap` closures that set `presentedLegalDoc`. The `.sheet(item: $presentedLegalDoc)` modifier stays on `WelcomeView`.
- The `revealStep` opacity gates on each button stay where they are now (in `WelcomeView`, applied to the new component instances). The new components don't know about reveal animation ÔÇö they just render.

## Accessibility

- All sizes use `@ScaledMetric(relativeTo: .body)` so the layout scales with Dynamic Type.
- The checkbox tap target is at least 44├Ś44pt regardless of the visual box size.
- The checkbox accessibility element is collapsed via `.accessibilityElement(children: .combine)` with `.isButton` and `.isSelected` traits.
- At very large accessibility text sizes (AX1ÔÇôAX5), the auth screen will exceed one viewport and require scrolling. We accept this ÔÇö it's standard iOS behavior, and forcing-fit content at AX sizes is the cause of most accessibility layout bugs.
- Pixel-perfect match holds at the default Dynamic Type size shown in the mockups; at other sizes everything scales proportionally.

## Risks

- **Touching #358 (recently shipped welcome screen).** Mitigation: the changes to `WelcomeView` are a pure swap-out of three inline views (Apple button, Google button, disclaimer Text) for shared components with identical visuals and identical action callbacks. The `revealStep` state, animation cadence, and the two top buttons stay byte-identical.
- **Global `UISegmentedControl.appearance()` configuration.** Mitigation: there is no other segmented Picker in the app today. If a second one is added, scope the appearance via `appearance(whenContainedInInstancesOf:)` ÔÇö the call site in `PebblesApp.init` is the single place to change.
- **Custom placeholder overlay on `PebblesTextInput`.** SwiftUI doesn't expose placeholder color directly. Risk: the overlay approach is more code than `.textFieldStyle(.roundedBorder)`. Acceptable given the spec requires placeholder color to be `pebblesMutedForeground`, not the system default.

## Out of scope

- Snapshot / pixel diff testing.
- Light/dark visual parity audits beyond a manual check.
- Animation polish on switcher mode change (default is fine).
- Any change to OAuth provider configuration, Supabase wiring, or `LegalDocumentSheet`.
- Updating the Arkaik bundle (no screens, routes, data models, or endpoints are added/renamed/removed; per `CLAUDE.md`, Arkaik updates are only required when product architecture changes).
