# M35 Naming Harmonization — Design

**Issue:** #487
**Milestone:** M35 · Features and Model consistency
**Status:** Design approved; awaiting plan

## Goal

Harmonize the model-layer vocabulary across iOS, web, admin, Supabase, and edge functions for the concepts listed in #487 — but only where the harmonization pays off. The audit in #487 is the authoritative inventory; this design is the *strategy* layered on top of it: what's in scope, in what order, and at what depth (code-only vs DB-deep).

## Decisions

### Vocabulary destinations

| Concept | Final dev-code name | Final DB-column name |
|---|---|---|
| User-drawn SVG mark | `glyph` / `Glyph` | `glyph_id` (already) |
| Composed visual (glyph + base shape) | — *(not introduced in M35)* | — |
| ±1 numeric axis | `polarity` (replaces `positiveness`) | `positiveness` *(unchanged in M35)* |
| 1–3 numeric scale | `intensity` (already) | `intensity` (already) |
| 9-case combination (polarity × intensity) | `Valence` (already correct in iOS, fixed in web/edge) | — |
| 3-case string label (`lowlight`/`neutral`/`highlight`) | `PolarityKey` (unified) | — |
| 3-case string label for intensity | `IntensityKey` (unified) | — |
| Pebble pictures | `snap` (already in types and DB) | `snaps` (already) |
| People / pets / entities | `soul` (already in types and DB) | `souls` (already) |

### Strategic choices that shape the plan

**1. The `positiveness` → `polarity` rename is code-only in M35 (Tier C).**
The DB column stays `positiveness` throughout this milestone. Each platform translates at its boundary:

- Web: `supabase-provider.ts` maps `row.positiveness ↔ domain.polarity`.
- iOS: `CodingKeys` on every struct that decodes pebble rows (`Valence`, `Pebble`, `PebbleDetail`, `ComposePebbleResponse`, `PebbleCreatePayload`, `PebbleUpdatePayload`) maps JSON key `positiveness` to Swift property `polarity`. Raw PostgREST select strings keep saying `positiveness`.
- Edge functions: a small domain mapper in `_shared/db-mappers.ts` exposes a `pebbleFromRow`/`pebbleToRow` boundary so engine code only ever sees `polarity`.

**Rationale.** The audit treats `positiveness` → `polarity` as the only **critical-risk** item — it requires a coordinated multi-platform DB migration, iOS dual-read, version gating of old clients, RPC recreation across 8 migration files, and an edge-fn redeploy. The benefit of the column rename is purely visible in raw SQL and the autogenerated `database.ts` — neither read by anyone outside DBA-level work.

The author's justification for the rename (avoid value-stigmatizing language; `positive` carries directional weight) is fully satisfied at the dev-code layer. The DB column is an implementation detail. Fold the column rename into a future pebbles-touching migration PR at zero marginal coordination cost.

**Tension acknowledged.** The audit elsewhere flags the `mark_id` boundary alias as a smell to remove. PR 1 of this design does remove it. PR 3 reintroduces an analogous alias for `positiveness`. They are not equivalent: Mark→glyph is web-only with zero migration cost; positiveness→polarity is cross-platform with critical migration cost. This is a strategic exception, not a principled reversal.

**2. Petroglyph is out of scope for M35.**
Every other audit item is a rename of an existing concept; petroglyph is the only *additive* concept. The rendering engine has no current pain point that an explicit `Petroglyph` type would resolve. Introduce only if a future change in the rendering pipeline benefits from it.

**3. Snaps analytics column rename (`pct_with_picture` → `pct_with_snap`) is deferred.**
Cost is a migration plus admin component churn; benefit is one internal column name match. Bundle into a future analytics-migration PR. UI label and i18n key updates are in scope.

**4. Souls onboarding copy is NOT updated.**
The model/UI divergence in `WelcomeSteps.swift` ("people" instead of "souls") is intentional brand-voice translation. Updating it would push internal jargon into end-user copy. The audit's recommendation here is rejected.

**5. The 3-case string-form types unify under `PolarityKey` and `IntensityKey`.**
Across platforms today these are named `Polarity` (web), `ValencePolarity` (iOS), `PebbleValence` (edge — misnamed) for polarity, and `ValenceSizeGroup` (iOS), `PebbleSize` (edge) for intensity. Unify on `PolarityKey` / `IntensityKey` to make their relationship to the numeric concepts explicit and to free the name `Polarity` for the numeric type in PR 3.

## PR sequence

### PR 1 — Glyph rename (web only)

Independent, low risk. Can ship in parallel with anything else.

- Rename `Mark` type → `Glyph` in `lib/types.ts`.
- Rename data-provider interface methods (`createMark`, `updateMark`, `listMarks`, `getMark`, `deleteMark`, `CreateMarkInput`, `UpdateMarkInput`) to `Glyph` equivalents.
- Rename hooks: `useMark` → `useGlyph`, `useMarks` → `useGlyphs`, `markMap` → `glyphMap` in `useLookupMaps.ts`.
- Rename `mark` prop to `glyph` in `GlyphCard`, `GlyphDetail`, `GlyphPreview`, `PebbleVisual`, `SoulGlyphThumbnail`, `CarveEditor`.
- Rename `mark` param to `glyph` in `lib/engine/params.ts` `extractGlyph`.
- Remove the `mark_id ↔ glyph_id` boundary alias in `supabase-provider.ts`; use `glyph_id` directly.
- Update `lib/data/karma.ts` to reference `pebble.glyph_id`.

Approximate touch: ~12 web files. No DB, no iOS, no edge.

### PR 2 — 3-case key type unification + valence cleanup

Cross-platform but conceptually unified. One PR.

**Web** (`apps/web/lib/config/emotion-category-ordering.ts`, `components/record/ValenceIntensityGrid.tsx`):
- Rename type `Polarity` (the string union `"lowlight"|"neutral"|"highlight"`) → `PolarityKey`.
- The name `Polarity` is now free. Introduce `type Polarity = -1 | 0 | 1` as the new numeric type alias.
- Remove the incorrect type alias `Valence = -1 | 0 | 1`; its callers retype as `Polarity` (numeric). The field on row/domain types remains *named* `positiveness` in PR 2 — only the type changes here. The field-name rename happens in PR 3.
- Update `POLARITIES`, `SHAPE_POLARITY` locals in `ValenceIntensityGrid.tsx` accordingly.

**iOS** (`apps/ios/Pebbles/Features/Path/Models/Valence.swift` and callers in `Path/ValencePickerSheet.swift`, `Render/PebbleAnimatedRenderView.swift`, `Render/PebbleOutlineBackdropView.swift`):
- Rename `ValencePolarity` enum → `PolarityKey`.
- Rename `ValenceSizeGroup` enum → `IntensityKey`.
- Rename computed property `var polarity: ValencePolarity` → `var polarityKey: PolarityKey`.
- Rename computed property `var sizeGroup: ValenceSizeGroup` → `var intensityKey: IntensityKey`.
- Update tests: `PebblesTests/ValenceMetadataTests.swift`, `PebbleOutlineGeometryTests.swift`, `EmotionCategoryOrderingTests.swift`.

**Edge** (`packages/supabase/supabase/functions/_shared/engine/`):
- Rename `PebbleValence` → `PolarityKey` in `types.ts`.
- Rename `PebbleSize` → `IntensityKey` in `types.ts`.
- Rename `intensityToSize()` → `intensityToKey()` in `resolve.ts`.
- Update cascading references in `layout.ts`, `shapes/index.ts`.
- **Do not rename `positivenessToValence()` in this PR** — its input parameter is still called `positiveness` here; it gets renamed in PR 3 alongside the input rename.

### PR 3 — `positiveness` → `polarity` code-only rename (Tier C)

Depends on PR 2. Largest of the PRs. Could be split per platform (3a web, 3b iOS, 3c edge) if review surface becomes too large — each platform is independent because the wire format (`positiveness` on PostgREST) is stable. Default: one PR; split if reviewer requests.

**Web:**
- Rename `positiveness` field on every type in `lib/types.ts`.
- Rename file `lib/config/positiveness.ts` → `lib/config/polarity.ts`; rename `POSITIVENESS_SIGNS` → `POLARITY_SIGNS`, `POSITIVENESS_KEY` → `POLARITY_KEY` if present.
- Rename component `PositivenessIndicator` → `PolarityIndicator` and file accordingly.
- Update all references in `lib/engine/params.ts`, `render.ts`, `templates.ts`, `types.ts`, components under `record/`, etc.
- In `lib/data/supabase-provider.ts`: introduce a `positiveness ↔ polarity` boundary translation at every read/write site for pebbles. Pattern: row mappers `pebbleFromRow`/`pebbleToRow`, parallel to the (now removed) Mark/Glyph alias.
- The `Polarity` type alias already exists from PR 2; PR 3 only renames the field that uses it.

**iOS:**
- Rename Swift property `Valence.positiveness` → `polarity`.
- Add `CodingKeys` on every struct that decodes pebble rows to map JSON `positiveness` ↔ Swift `polarity`: `Valence`, `Pebble`, `PebbleDetail`, `ComposePebbleResponse`, `PebbleCreatePayload`, `PebbleUpdatePayload`.
- **Leave raw PostgREST select strings unchanged** — they still say `positiveness` because the DB column is unchanged. Add a single-line comment on each such site noting the deliberate deviation, scheduled for cleanup when the DB column renames.
- Update all in-app field access (`pebble.positiveness` → `pebble.polarity`).

**Edge:**
- Create `_shared/db-mappers.ts` with `pebbleFromRow(row): PebbleDomain` and `pebbleToRow(domain): PebbleRowInsert` covering the `positiveness ↔ polarity` field. (Future: extend with any other DB-only legacy names that arise.)
- Update all `pebble.positiveness` references in `compose-and-write.ts`, `engine/resolve.ts`, `engine/layout.ts`, `shapes/index.ts` to consume the domain object.
- Rename `positivenessToValence()` → `polarityToKey()`.

**Discipline boundary.** After PR 3, `database.ts` (regenerated from the schema) still types the row's column as `positiveness`. The boundary mappers are the *only* place this name appears in TypeScript. Direct row access bypassing the provider/mapper is a code smell; reviewers should flag. Document this in `docs/agents/data-and-async.md` as part of PR 3.

### PR 4 — Snaps UI labels + i18n (web only)

Independent. Tiny PR.

- `PebbleDetailAddToolbar.tsx`: i18n key `picture` → `snap`, `pictureAria` → `snapAria`.
- `PebbleDetail.tsx`: i18n key `photoAlt` → `snapAlt`.
- Update web translation files for `en` and `fr`.

## Sequence and dependencies

```
PR 1 (Glyph)        ─────────┐
PR 4 (Snaps i18n)   ─────────┤
                             │
PR 2 (PolarityKey/IntensityKey)  ──→  PR 3 (positiveness → polarity)
```

- PR 1, PR 4: any time, parallel.
- PR 2 → PR 3: PR 2 frees `Polarity` for use as the numeric type name; PR 3 then claims it.
- PR 3 may optionally split per platform (3a/3b/3c) if review surface is large.

## Deferred work

- **DB column rename** (`positiveness` → `polarity`): fold into the next migration PR that already touches the `pebbles` table for other reasons. At that point, drop the supabase-provider alias, iOS `CodingKeys` for this field, and edge mapper field.
- **Snaps analytics column rename** (`pct_with_picture` → `pct_with_snap`): fold into the next analytics-migration PR.
- **Petroglyph**: introduce as a named type if and when a rendering-engine change benefits from it.
- **`render_svg` → `petroglyph_svg`**: not currently justified.

## Out of scope (explicitly rejected)

- Updating `WelcomeSteps.swift` "people" → "souls" — model/UI divergence is intentional brand-voice translation.
- Updating privacy docs to use "souls" — same reason.
- Renaming `instants` (signed-URL projection on the read path) — this is a deliberate distinction from `snap` (the write entity), not an inconsistency.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Tier C boundary aliases drift out of sync as new code is added | Document the discipline boundary in `docs/agents/data-and-async.md`; reviewers flag direct row access |
| PR 3 web rename misses a `positiveness` reference | rg sweep before merge; CI compiles strictly typed code |
| iOS `CodingKeys` mapping forgotten on a new struct that decodes pebble rows | Add a checklist note in `apps/ios/CLAUDE.md` or equivalent |
| Reviewer fatigue on PR 3 size | Split into 3a/3b/3c per platform; each is independent under Tier C |
| Future DB rename misses a Tier C alias and breaks an unupdated layer | When the column-rename PR is opened, grep for `positiveness` across all platforms and confirm zero non-mapper references remain |

## Acceptance for M35

- All four PRs merged.
- `rg positiveness apps/ packages/` returns matches only inside (a) the supabase-provider boundary mapper, (b) iOS `CodingKeys` lines and raw PostgREST select strings, (c) edge `db-mappers.ts`, (d) regenerated `database.ts`, (e) migration files.
- No new code references `Mark`, `PebbleValence`, `PebbleSize`, `ValencePolarity`, `ValenceSizeGroup`, `positivenessToValence`, or `intensityToSize`.
- The 3-case label types are unified as `PolarityKey` and `IntensityKey` across all platforms.
